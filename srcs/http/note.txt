3. Status code
HTTP response status codes indicate whether a specific HTTP request has been successfully completed. Responses are grouped in five classes:

Informational responses (100 – 199)
Successful responses (200 – 299)
Redirection messages (300 – 399)
Client error responses (400 – 499)
Server error responses (500 – 599)

4. RFC semantic rules
		1. Header Case Insensitivity
		This is the trickiest part for your std::map. By default, std::map<std::string, std::string> treats "Host" and "host" as different keys.

		The Fix: Before inserting or looking up a key in your map, convert the header name to lowercase.

		Code logic: _headerMap[toLower(name)] = value;

		2. Mandatory Host Header
		The subject requires accurate status codes.

		The Fix: After your header loop finishes (when you see \r\n\r\n), check if _headerMap.count("host") is 0.

		Action: If it's missing, immediately set _statusCode = 400 and stop processing the request.

		3. Strict Whitespace (The "No Space Before Colon" Rule)
		This is a common "trick" in 42 projects to see if you read the RFC.

		The Rule: Field-Name ":" [Field-Value]. There is zero space allowed between the name and the colon.

		The Fix: In your while loop that reads the name, if you encounter a space ' ' before you find the ':', the request is malformed.

		Action: Set _statusCode = 400.


5. IP and Port
- Webserver ports
	communication endpoints that allow data to flow b/w client & server over the internet
	ensures incoming messages to be directed to the appropriate destination on the server e.g. website, email, application
	defined by a number ranging between 0-65535 -> often 80 or 443 to exchange content over the internet

	Data arrives at the computer's IP address -> Port gives the final destination label.
	the Host header is mandatory but the port is often omitted if it is the default.

	serverIP: ip address of the server that the client connected to
	clientIP: IP address of the person sending the request - for logging, where the request comes from


	when listening for a connection, it is listening t a specific PORT (specific folder)

	IP Address: Finds the computer.
	Server Port: Finds the specific "Virtual Server" or website on that computer.
	Connection: The temporary private tunnel between the user and that port.

	-> there are multiple servers being hosted via virtual hosting.


When a request comes in, it doesn't just hit "the webserver." It hits a specific listening socket on a specific Port.
The Arrival: A user types localhost:9090 in their browser.
The Entry: The request enters your program through the socket that was set up for Port 9090.
The Labeling: IO part sees this and should tell your HTTPrequests object: "Hey, this request belongs to Port 9090" (This is your Step 1 setter).
The Choice: When The Handler receives your completed request object, he calls req.getServerPort(). He sees 9090.
The Mapping: Because he knows it’s 9090, he goes to the config and finds Server Block 2. He now knows the "Destination" for this request is the /www/siteB folder.
